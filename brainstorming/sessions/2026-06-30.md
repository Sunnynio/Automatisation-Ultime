## 📋 **Décisions Prises pendant la Session**

### **1. Architecture Globale**
| Décision | Justification |
|---------|---------------|
| **Notion** est le **cœur du système** (base de données + interface visuelle). | Centralisation des données et flexibilité pour les widgets. |
| **Google Calendar** est utilisé pour le *Time Blocking* et la synchronisation. | Intégration native avec l’écosystème Google. |
| **Gemini** est l’**interface principale** (requêtes vocales/écrites, filtrage GPS). | Accès à Google Calendar/Gmail/Drive et compréhension du langage naturel. |
| **Mistral** est responsable de **l’automatisation et de l’exécution des tâches**. | Flexibilité pour les scripts Python et manipulation de Notion. |

---

### **2. Structure Notion**
| Décision | Justification |
|---------|---------------|
| **Master Board** : Une seule base pour toutes les tâches (pour commencer). | Simplicité et centralisation. |
| **Propriétés obligatoires** : `Nom`, `Durée`, `Support`, `Pays/Lieu`, `Statut`, `Priorité`, `Récurrence`. | Permet un filtrage contextuel de base. |
| **Propriétés recommandées** : `Contexte`, `Heure de la journée`, `Dépendances`, `Date Limite`. | Améliore la précision du filtrage. |
| **Propriétés pour l’automatisation** : `Délégable à l’IA`, `Statut IA`, `Livrable`, `Date de Délégation`. | Permet à Mistral d’exécuter et suivre les tâches. |
| **Propriétés pour la gamification** : `Date de Complétion`, `Temps Réel`, `Points`. | Motivation et suivi des progrès. |

---

### **3. Synchronisation**
| Décision | Justification |
|---------|---------------|
| Utiliser **Make.com** pour prototyper la synchronisation Calendar ↔ Notion. | No-code, facile à configurer. |
| **Sens de la synchronisation** : **Calendar → Notion** (pour commencer). | Éviter de surcharger Calendar. |
| Utiliser un **ID unique** (ex: `GCAL_{ID_EVENEMENT}`) dans Notion pour lier les entrées. | Éviter les doublons. |

---

### **4. Workflows**
| Décision | Justification |
|---------|---------------|
| **Filtre contextuel** : Algorithme de priorisation = **Priorité > Durée > Date Limite**. | Simple et efficace pour commencer. |
| **Nombre de suggestions** : **Top 3 tâches** par défaut. | Éviter la surcharge cognitive. |
| **Format de sortie** : **Tableau Markdown** (widgets) ou **liste vocalisée** (Google TTS). | Adapté à tous les supports. |
| **Gamification** : **10 points par 30 min de tâche terminée** (base). | Système simple et motivant. |
| **Résumé quotidien** : Généré par **Mistral** et envoyé par **email**. | Automatisation complète. |

---

### **5. Outils et Technologies**
| Décision | Justification |
|---------|---------------|
| **Langage principal** : **Python** pour les scripts d’automatisation. | Flexibilité et bibliothèques disponibles. |
| **Bibliothèques Python** : `notion-client`, `google-api-python-client`, `requests`. | Standard pour les API utilisées. |
| **Hébergement** : **Local** pour les tests, **Google Cloud Functions** pour la production. | Scalabilité et intégration avec Google. |

## ❓ **Points Ouverts et Questions**

### **1. Structure Notion**
- **Question** : Faut-il ajouter la propriété **"Énergie Requise"** (Faible/Moyenne/Élevée) ?
  - *Pour* : Permet un filtrage plus précis (ex: éviter les tâches "Élevée" si l’utilisateur est fatigué).
  - *Contre* : Complexifie la structure.
  - **Prochaine étape** : Tester avec des données fictives pour évaluer l’utilité.

- **Question** : Faut-il ajouter la propriété **"Type de Tâche"** (Administrative, Créative, etc.) ?
  - *Pour* : Permet un filtrage par type d’activité.
  - *Contre* : Peut être redondant avec `Contexte`.
  - **Prochaine étape** : Discuter avec d’autres IA (Gemini) pour avoir leur avis.

---

### **2. Synchronisation**
- **Question** : Faut-il synchroniser **Notion → Calendar** (en plus de Calendar → Notion) ?
  - *Pour* : Centralisation complète.
  - *Contre* : Risque de surcharger Calendar.
  - **Prochaine étape** : Tester la synchronisation unidirectionnelle d’abord.

- **Question** : Quel outil utiliser pour la synchronisation à long terme : **Make.com** ou **scripts Python** ?
  - *Make.com* : Plus simple, mais limité aux connecteurs disponibles.
  - *Scripts Python* : Plus flexible, mais nécessite de l’hébergement.
  - **Prochaine étape** : Prototyper avec Make.com, puis évaluer les limites.

---

### **3. Automatisation**
- **Question** : Comment Mistral peut-il **savoir quelles tâches il peut exécuter** ?
  - *Option 1* : Utiliser un **système de tags** (ex: `#email`, `#analyse`).
  - *Option 2* : Créer une **liste prédéfinie** de types de tâches.
  - *Option 3* : Laisser Mistral **deviner** en fonction du `Nom` et du `Contexte`.
  - **Prochaine étape** : Implémenter l’option 1 (tags) et tester.

- **Question** : Faut-il permettre à Mistral de **modifier Google Calendar** ?
  - *Pour* : Automatisation complète (ex: planifier une réunion).
  - *Contre* : Risque de sécurité.
  - **Prochaine étape** : Commencer par **ne pas permettre** les modifications, puis évaluer le besoin.
